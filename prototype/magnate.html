Prototype. This runs Magnate's economy model (uniform-price batch auction with reference-qualification, 4-tier supply chain, an NPC market-maker desk, retail-demand faucet + sinks, B2B contracts with escrowed bonds, and a conservation invariant over all four system accounts — WORLD_FAUCET, SINK_BURN, NPC_DESK, CONTRACT_ESCROW — plus firm cash) live in your browser — the same model verified in play_magnate.py. It is not the production React + TigerBeetle/Temporal/Nakama client; it is a faithful, deterministic visualization of the design. Balance is illustrative (real tuning is Epic 9's job).

About this prototype

You run PLAYER, a company in a shared, player-driven economy against 10 NPC rivals. Source raw materials, run multi-tier production, and trade on a uniform-price batch-auction market whose prices emerge from real supply/demand plus price-elastic retail consumers. Specialize and out-think the market — there is no combat, no twitch.

  • The market is the game — one uniform clearing price per product each tick; no latency/geography edge.
  • Legible economy — every number traces to a clear; ∑ conserved proves money is never created/destroyed outside the named accounts (WORLD_FAUCET, SINK_BURN, + firm cash).
  • Specialization beats sprawl — pick a product, dominate it; the retail-facing T3 goods (phone/car) carry the demand.

Honesty: this is a browser visualization running the same model + formulas as play_magnate.py (deterministic seeded RNG; run it: python3 play_magnate.py), conservation-checked every tick. Vs. the full design: it now models all four ledger accounts (WORLD_FAUCET retail mint, SINK_BURN sinks, NPC_DESK market-maker desk, CONTRACT_ESCROW B2B bonds); charts include candlestick+volume (market), a supply-chain Sankey, a net-worth area, and inventory bars (utilization gauges/heatmaps are not modeled); reference-qualification and manual production planning ARE modeled; the sim is user-driven (Play/Pause/Step, honoring reduced-motion). It is not the production client (React + TigerBeetle/Temporal/Nakama/Postgres/Valkey), which needs the real backend. Economy balance is illustrative; real tuning is the offline Mesa/SimPy work (Epic 9). See the planning docs in _bmad-output/.
